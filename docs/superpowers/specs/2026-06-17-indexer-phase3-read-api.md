# Indexer Phase 3 — TS Drizzle Read-Only Mirror + Hono REST API

> Date: 2026-06-17
> Depends on: Phase 1 (ConfigCreated event), Phase 2 (Rust ingest crate, live schema in `creatorflow-pg`)
> Authoritative API contract: `docs/superpowers/specs/2026-06-15-indexer-design.md` §"Read API (REST, Hono)"

## Goal

Expose the indexer's Postgres tables (written by the Rust `sui-indexer-alt-framework`
ingest) as a read-only REST API for the dashboard. TypeScript layer is a **read-only
mirror** — Rust owns ingest + the diesel migration (single source of truth). Drizzle
schema is generated by `drizzle-kit pull`, never hand-authored, never migrates.

## Decisions (locked)

- **Runtime / toolchain**: Node + `tsx` + `pnpm`. Hono on `@hono/node-server`.
- **u64 serialization**: all `BIGINT` (amounts, versions, timestamps) returned as
  JSON **strings** — avoids JS `Number > 2^53` precision loss. Dashboard parses to
  BigInt as needed.
- **Scope**: all 7 endpoints (no MVP subset).
- **Location**: `api/creatorflow-api/` (sibling to `indexer/`, `move/`).
- **Pagination**: keyset (not offset); opaque base64 cursor. Cursor tuple MUST be
  unique per row — for the 4 event tables that is `(timestamp_ms, tx_digest, event_seq)`
  (a single PTB can emit multiple same-type events: same `tx_digest`, different
  `event_seq`, possibly identical `timestamp_ms` — a 2-tuple cursor would skip/dup
  that batch at the page boundary). `config_created` is keyed by `config_id` (already
  unique) so `(checkpoint_timestamp_ms, config_id)` suffices.
- **DB**: connect to existing `creatorflow-pg` (host port **5433**), read-only role.
  No migrations from TS.

## §1 Architecture

```
api/creatorflow-api/
  src/
    db/
      schema.ts        # drizzle-kit pull output (read-only mirror, do not hand-edit)
      client.ts        # drizzle(node-postgres pool), read-only role
    lib/
      cursor.ts        # encode/decode opaque base64 keyset cursor (pure)
      serialize.ts     # u64 BIGINT -> string serialization, null handling (pure)
    routes/
      configs.ts       # list / detail / splits / mutations / summary
      collaborators.ts # earnings (JOIN)
      vaults.ts        # withdrawals (LEFT JOIN)
    app.ts             # Hono instance + route mount + onError handler
    server.ts          # @hono/node-server entry
  drizzle.config.ts    # pull config (points at 5433)
  tests/*.test.ts      # vitest: seed existing pg -> hit endpoints
  package.json / tsconfig.json
```

Boundaries: Drizzle layer queries only (never writes); routes compose queries +
serialize; `lib/` is pure and independently testable. Each route file = one resource
group.

## §2 Endpoints & query shapes

All list endpoints use keyset pagination; opaque base64 cursor. Cursor tuple = the
table's unique ordering key (see Decisions): 3-tuple `(ts, tx_digest, event_seq)` for
event tables, `(ts, config_id)` for `config_created`. `limit` default 50, clamp max 200.

| Endpoint | Query | Sort / pagination |
|---|---|---|
| `GET /configs?owner=&cursor=&limit=` | `config_created WHERE owner=$1` | keyset `(checkpoint_timestamp_ms DESC, config_id DESC)`; owner normalized lowercase |
| `GET /configs/:id` | `config_created` + subquery `config_mutated ORDER BY new_version DESC LIMIT 1` for latest version | single; 404 if absent |
| `GET /configs/:id/splits?cursor=&limit=` | `split_executed WHERE config_id` + embed `recipient_payout` (single `IN` backfill, no N+1) | keyset `(timestamp_ms DESC, tx_digest DESC, event_seq DESC)` |
| `GET /configs/:id/mutations?cursor=&limit=` | `config_mutated WHERE config_id` | keyset `(checkpoint_timestamp_ms DESC, tx_digest DESC, event_seq DESC)` |
| `GET /configs/:id/summary` | `split_executed` agg: `SUM(amount_in/tax/savings/protocol_fee/yield), COUNT(*)` | single aggregate; empty -> all zeros |
| `GET /collaborators/:addr/earnings?cursor=&limit=` | `recipient_payout JOIN split_executed USING (tx_digest,event_seq) WHERE recipient=$1`, returns payout + split context (config_id, timestamp) | keyset on split `(timestamp_ms DESC, tx_digest DESC, event_seq DESC)`; addr normalized |
| `GET /vaults/:id/withdrawals?cursor=&limit=` | `vault_withdrawn LEFT JOIN config_created ON vault_id IN (tax_vault_id, savings_vault_id)` | keyset `(checkpoint_timestamp_ms DESC, tx_digest DESC, event_seq DESC)`; config_id may be null |

> Why 3-tuple for event tables: one PTB can emit multiple same-type events (batched
> `execute_split`, or withdraw tax + savings in one tx) → same `tx_digest`, different
> `event_seq`, possibly identical `timestamp_ms`. A 2-tuple `(ts, tx_digest)` boundary
> would skip or duplicate that batch. `event_seq` is the PK tie-breaker that restores
> uniqueness. `collaborator earnings` orders by the JOINed `split_executed` key, so it
> carries `event_seq` too.

Key points:
- **Splits payout backfill**: fetch the page of N splits (with `(tx_digest,event_seq)`),
  then one `recipient_payout WHERE (tx_digest,event_seq) IN (...) ORDER BY payout_idx`,
  group in code. Explicit `IN` over SQL JSON agg — more controllable and testable.
- **LEFT JOIN (vaults)**: per §"Read API" of the design spec — a withdrawal whose
  `config_created` row has not yet been written must not vanish; `config_id` is null
  until it catches up.
- **Address normalization**: query params lowercased before comparison; Rust ingest
  already stores normalized hex — consistency covered by tests.

## §3 Error handling

- Uniform error shape `{ error: { code, message } }` via Hono `app.onError`.
- **400** — invalid query: bad cursor (base64 decode fail / format mismatch),
  non-numeric `limit`, `owner`/`addr` not `^0x[0-9a-f]{1,64}$` after normalization.
  Validate at route entry (zod or hand-written).
- **404** — `GET /configs/:id` not found. Other list queries empty -> **200 empty
  array** (an empty result is a valid state, not an error).
- **500** — DB failure: full log, generic message to client (no SQL/schema leak).
- **Read-only defense**: DB role itself is read-only (a bug still cannot write);
  connection pool failure -> fail loud at startup, no silent degradation.
- **No auth** (public read demo); `limit` clamp + cursor validation block
  amplification. CORS open to dashboard origin.

## §4 Testing (vitest)

Against existing `creatorflow-pg` (5433): each suite `beforeEach` TRUNCATEs the 5
tables -> seeds fixture -> hits endpoint and asserts.

- **Per-endpoint happy path**: correct rows, field types (u64 as string), correct JOINs.
- **Pagination boundaries**: seed > limit rows -> page 1 returns cursor, page 2
  continues with no overlap/no gap, last page cursor=null.
- **Cross-config aggregation**: one collaborator across multiple configs -> earnings
  captures all.
- **Unit (lib)**: `cursor.ts` encode∘decode round-trip + bad input; `serialize.ts`
  bigint/null.
- **Monkey (test.md, mandatory)**:
  - empty `recipient_payout` for a split -> splits payout=[]
  - zero amount -> serialized `"0"`, not `0`/null
  - same addr multiple payouts in one split -> earnings not undercounted
  - out-of-order checkpoint arrival (high ts seeded first) -> keyset order still correct
  - `vault_withdrawn` with no matching config_created -> config_id null (LEFT JOIN)
  - **multiple same-type events in one tx** (same `tx_digest`, different `event_seq`,
    identical `timestamp_ms`) straddling a page boundary -> no row skipped or
    duplicated (directly exercises the 3-tuple cursor)
  - very long history paginates to the end without blowup

Verification gate: `pnpm test` green + `npx tsc --noEmit` clean. (No git repo — skip
commit gate; `sui move test` unaffected.)
