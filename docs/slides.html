The problem

Splitting revenue is a monthly
chore of spreadsheets and trust.

A payment lands. Now someone manually pays the editor, the co-writer, sets aside tax, moves a little into savings. It is error-prone, it relies on trust — and the savings that do get set aside just sit there earning nothing.

The solution

One on-chain rule. One transaction.

A creator defines a split config once. After that, any incoming USDC payment is split atomically — conserved to zero dust.

Why Sui

Access control at the type level.

Each vault is guarded by its own capability object. Compromise of one never cascades — and reusing the wrong capability is rejected before execution, by Sui's type system.

// a creator's TaxCap aimed at a *different* config's vault
vaults::withdraw_tax(taxCap_A, taxVault_B, …)
  🛑 abort EWrongCap cap.vault_id != vault.id

vaults::withdraw_tax(taxCap_A, taxVault_A, …)
  ✓ ok un-forgeable, per-object blast radius
